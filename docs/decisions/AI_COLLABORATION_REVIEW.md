# AI Collaboration Direction Review

## 1. 결론

현재 정리한 방향은 긍정적이다.

다만 조건이 있다.

```text
AI를 거부하지 않는다.
AI에게 전부 위임하지도 않는다.
사용자가 통제권을 유지한 상태에서 AI를 설계 보조자, 코드 초안 작성자, 오류 분석자, 리뷰어로 사용한다.
```

이 방향은 생산성만 최우선으로 하는 방식보다 느리다. 하지만 교육자이자 초급~중급 전환 단계의 개발자에게는 더 건강한 방식이다.

## 2. 왜 긍정적인가

### 2-1. 도구 변화에 강하다

AI 도구는 계속 바뀐다.

```text
ChatGPT
Codex
Cursor
Claude Code
기타 IDE Agent
```

특정 도구가 바뀌어도 남는 능력은 다음이다.

```text
요구사항 정리
데이터 구조 이해
권한 설계
보안 판단
코드 리뷰
테스트
배포 흐름 이해
오류 원인 분석
```

사용자가 전체 프로세스를 통제하는 방식은 이 능력을 남긴다.

### 2-2. 유지보수 가능성이 높다

AI가 만든 코드를 이해하지 못하면 앱이 작동해도 유지보수가 어렵다.

이 프로젝트의 방식은 다음을 요구한다.

```text
코드를 직접 읽는다.
변경 이유를 확인한다.
실행 결과를 확인한다.
테스트를 통과시킨다.
다음 단계로 넘어가는 기준을 둔다.
```

따라서 유지보수 가능성이 높다.

### 2-3. 교육적으로 가치가 있다

학생들에게 필요한 것은 “프롬프트 한 번으로 앱 만들기”가 아니다.

필요한 것은 다음 능력이다.

```text
문제를 설명하는 능력
요구사항을 나누는 능력
AI의 코드를 검토하는 능력
오류 메시지를 읽는 능력
테스트 결과를 해석하는 능력
자신이 만든 결과물을 설명하는 능력
```

현재 방향은 이 능력을 키우는 데 적합하다.

### 2-4. 사용자의 현재 수준과 맞다

사용자는 Flask, Docker, Git, WSL 등에 대한 경험은 있지만, Flask + PostgreSQL + Alembic + OAuth + OCI 전체를 혼자 설계하고 구현할 수준은 아직 아니다.

따라서 다음 두 극단은 모두 적절하지 않다.

```text
전부 AI에게 맡김
전부 혼자 함
```

가장 적절한 방식은 다음이다.

```text
사용자가 주도한다.
AI가 단계별로 돕는다.
사용자가 이해 가능한 단위로 적용한다.
막히면 AI가 오류 분석과 대안을 제시한다.
```

## 3. 위험 요소

이 방식에도 위험은 있다.

### 3-1. 속도가 느릴 수 있다

자동 생성 방식보다 느리다.

대응:

```text
학습 가치가 큰 초기 구조는 직접 진행한다.
반복 CRUD나 테스트 보강은 Codex 보조를 적극 사용한다.
```

### 3-2. 과도하게 모든 것을 이해하려 하면 지칠 수 있다

모든 라이브러리 내부까지 파고들 필요는 없다.

대응:

```text
프로젝트 유지보수에 필요한 수준까지만 이해한다.
코드 한 줄 한 줄 과잉 해설하지 않는다.
파일 역할, 데이터 흐름, 권한 흐름, 테스트 흐름을 우선 이해한다.
```

### 3-3. AI 사용량이 늘어날 수 있다

대화 기반 협업은 사용량이 많아질 수 있다.

대응:

```text
Pro는 설계/권한/보안/DB/배포/최종검토에만 사용한다.
중간/높음 모델은 일반 구현 검토와 오류 분석에 사용한다.
Codex는 반복 작업과 테스트 실행 보조에 사용한다.
```

### 3-4. “통제”가 “직접 다 하기”로 변질될 수 있다

통제권을 가진다는 것은 모든 코드를 혼자 작성한다는 뜻이 아니다.

정확한 의미는 다음이다.

```text
무엇을 바꾸는지 안다.
왜 바꾸는지 안다.
어떻게 검증하는지 안다.
최종 반영 여부를 사용자가 결정한다.
```

## 4. 냉정한 평가

이 방향은 “빠른 앱 납품”에는 최적이 아니다.

하지만 다음 목적에는 매우 적합하다.

```text
개발 프로세스 학습
AI 협업 개발 능력 형성
학생 수업 확장
포트폴리오 설명력 확보
장기 유지보수 능력 확보
```

따라서 이 프로젝트에서는 올바른 방향이다.

단, 모든 단계를 너무 천천히 할 필요는 없다. 다음처럼 구분해야 한다.

```text
직접 이해해야 하는 것:
- 프로젝트 구조
- DB 모델
- 권한 흐름
- 예약 충돌 로직
- CSV all-or-nothing
- 보안 정책
- Docker Compose 흐름

AI/Codex에 맡겨도 되는 것:
- 반복 CRUD 초안
- 비슷한 테스트 케이스 추가
- import 정리
- 문서 형식 정리
- 단순 Tailwind 조정
```

## 5. 최종 판단

사용자가 말한 “모든 프로세스를 내 통제하에 두는 것”은 AI 시대에 타당한 방향이다.

다만 이 말은 다음처럼 해석해야 한다.

```text
모든 코드를 직접 손으로 작성한다는 뜻이 아니다.
모든 의사결정과 검증 책임을 사용자가 가진다는 뜻이다.
```

이 기준이면 AI는 위협이 아니라 강력한 협업 도구가 된다.

이 프로젝트는 그 방식을 실험하고 정리하기에 좋은 사례다.


## 6. 이번 uv 전환에 대한 판단

`requirements.txt`에서 `uv + pyproject.toml + uv.lock`으로 전환한 것은 이 프로젝트 방향과 맞다.

이유:

```text
단순 설치 명령을 외우는 수준을 넘어서 의존성 선언, 잠금, 재현 가능한 실행 흐름을 이해할 수 있다.
Docker/OCI 배포 단계에서 로컬과 서버의 의존성 차이를 줄일 수 있다.
학생들에게도 '왜 내 컴퓨터에서는 되는데 서버에서는 안 되는가'라는 문제를 설명하기 좋다.
```

주의할 점:

```text
uv 내부를 처음부터 깊게 파고들 필요는 없다.
처음에는 uv init, uv add, uv add --dev, uv run, uv sync의 역할만 이해하면 충분하다.
의존성 관리 도구가 목적이 아니라 프로젝트 재현성과 유지보수성이 목적이다.
```


## 7. 점진적 구조화에 대한 판단

이번 대화에서 추가된 “처음부터 리팩토링을 감안하되, 처음부터 과하게 구조화하지 않는다”는 방향은 긍정적이다.

이 방향은 다음 두 극단을 모두 피한다.

```text
극단 1: 처음부터 복잡한 아키텍처를 만든다.
극단 2: 한 파일에 계속 기능을 쌓고 나중에 한 번에 정리한다.
```

좋은 흐름은 다음이다.

```text
한 파일로 작동시킨다.
불편함을 경험한다.
필요한 만큼 함수와 파일로 나눈다.
테스트한다.
다음 기능으로 간다.
```

교육적으로도 이 방식이 더 좋다.

이유:

```text
학생들이 구조화의 필요성을 체감한다.
리팩토링이 멋이나 유행이 아니라 문제 해결임을 이해한다.
AI가 제안하는 구조를 맹목적으로 받아들이지 않게 된다.
코드를 읽고 리뷰하는 능력이 남는다.
```

단, 주의할 점도 있다.

```text
너무 오래 단일 파일에 머물면 리뷰를 포기하게 된다.
너무 일찍 계층을 나누면 흐름을 이해하지 못한다.
```

따라서 이 프로젝트의 적정 기준은 다음이다.

```text
초기에는 단순하게 시작한다.
각 Phase 종료 시 Phase Maintenance를 수행한다.
리뷰가 어려워지는 순간 최소 범위로 구조화한다.
구조화 후 반드시 테스트한다.
```

냉정하게 보면, 이 방향은 생산성만 보면 느릴 수 있다. 그러나 사용자가 전체 프로세스를 이해하고, AI 도구가 바뀌어도 유지보수하며, 나중에 학생들에게 설명할 수 있는 개발 방식을 만드는 데에는 매우 적합하다.
## Git 기록과 사용자 통제

사용자 통제형 AI 협업 개발에서 Git 기록은 중요하다.

이유:

```text
AI 또는 Codex가 제안한 변경을 사용자가 검토했다는 흔적을 남길 수 있다.
Phase별로 어떤 판단을 했는지 추적할 수 있다.
나중에 학생들에게 개발 과정 전체를 보여줄 수 있다.
툴이 바뀌어도 프로젝트 변경 흐름을 사람이 이해할 수 있다.
```

따라서 이 프로젝트는 `[Phase] type: 한국어 설명` 형식의 커밋 메시지를 사용한다.
