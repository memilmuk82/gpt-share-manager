# Progressive Structuring and Phase Maintenance

## 1. 문서 목적

이 문서는 `gpt-share-manager` 프로젝트와 이후 수업형 프로젝트에서 사용할 개발 진행 원칙을 정의한다.

핵심은 다음이다.

```text
처음부터 완성형 아키텍처를 강제하지 않는다.
작동하는 코드에서 시작한다.
복잡해지는 순간을 관찰한다.
필요가 생겼을 때 최소 범위로 구조화한다.
각 Phase가 끝날 때 리뷰 가능한 상태로 정리한다.
```

이 방식은 생산성만을 극대화하기 위한 방식이 아니다. 사용자가 코드 흐름, 요구사항, 테스트, 유지보수 과정을 이해하면서 개발하기 위한 방식이다.

---

## 2. 왜 처음부터 복잡한 구조로 시작하지 않는가

초기부터 아래와 같은 구조를 강제하면 코드의 흐름보다 폴더 구조가 먼저 보인다.

```text
models/
services/
repositories/
routes/
schemas/
forms/
```

이 구조가 필요한 이유를 경험하지 못한 상태에서는 학습자가 구조를 이해하기보다 외우게 된다.

이 프로젝트는 다음 흐름을 우선한다.

```text
1. 단순한 코드로 작동시킨다.
2. 코드가 길어지거나 책임이 섞이는 불편함을 확인한다.
3. 그 불편함을 해결하기 위해 함수를 나눈다.
4. 다시 파일을 나눈다.
5. 필요할 때 route, service, model, test 구조로 확장한다.
```

수업용 표현으로는 다음 문장을 사용한다.

```text
작동하는 코드에서 유지보수 가능한 코드로 이동한다.
```

## 3. 점진적 구조화 원칙

### 3-1. 허용되는 시작점

초기 Phase에서는 단순 구조를 허용한다.

예:

```text
app.py 하나로 최소 Flask 실행
단순 route 함수
단순 health check
```

이 단계에서 복잡한 구조를 미리 도입하지 않는다.

### 3-2. 구조화가 필요한 신호

아래 신호가 보이면 구조화를 검토한다.

```text
한 파일이 너무 길어져 리뷰하기 어렵다.
하나의 파일이 여러 책임을 가진다.
같은 코드가 2회 이상 반복된다.
함수가 화면 처리, DB 처리, 권한 처리를 동시에 한다.
테스트를 작성하기 어렵다.
다음 기능을 추가하기가 두렵다.
오류 원인을 찾기 위해 너무 많은 코드를 훑어야 한다.
```

### 3-3. 구조화의 순서

구조화는 큰 설계 패턴을 한 번에 넣는 방식이 아니라 다음 순서로 진행한다.

```text
1. 긴 코드를 함수로 나눈다.
2. 반복 코드를 공통 함수로 뺀다.
3. route가 길어지면 Blueprint로 분리한다.
4. DB 모델이 생기면 models로 분리한다.
5. 비즈니스 규칙이 길어지면 service로 분리한다.
6. 테스트가 어려우면 테스트 가능한 단위로 다시 나눈다.
```

### 3-4. 금지되는 구조화

다음은 금지한다.

```text
필요성이 확인되지 않은 Repository 패턴 도입
학습자가 이해하지 못한 상태의 과도한 추상화
테스트 없이 파일 이동
기능 추가와 대규모 리팩토링을 한 번에 진행
AI가 제안한 복잡한 구조를 검토 없이 수용
```

## 4. Phase Maintenance

이 프로젝트에서는 “나중에 한 번에 리팩토링”하지 않는다.

각 Phase가 끝나면 짧은 유지보수 점검을 수행한다. 이를 `Phase Maintenance`라고 부른다.

### 4-1. Phase Maintenance 목적

```text
기능을 늘리기 전에 현재 코드를 리뷰 가능한 크기로 유지한다.
다음 Phase에서 수정하기 쉬운 상태로 만든다.
테스트와 문서를 현재 상태에 맞춘다.
```

### 4-2. Phase Maintenance 체크리스트

각 Phase 종료 시 아래를 확인한다.

```text
□ 기능이 의도대로 동작하는가?
□ pytest 또는 해당 Phase 검증이 통과했는가?
□ 단일 파일이 과도하게 길어지지 않았는가?
□ 하나의 파일이 여러 책임을 갖고 있지 않은가?
□ 함수 하나가 너무 많은 일을 하지 않는가?
□ 중복 코드가 반복되고 있지 않은가?
□ 권한/보안 검증 위치가 명확한가?
□ 테스트하기 어려운 구조가 생기지 않았는가?
□ TODO가 방치되지 않았는가?
□ PROJECT_STATUS.md가 갱신되었는가?
□ 필요하면 DEVELOPMENT_LOG.md 또는 결정 문서가 갱신되었는가?
```

### 4-3. 코드 크기 기준

절대 규칙은 아니지만 다음 기준을 사용한다.

```text
파일 300줄 이상:
- 책임이 섞였는지 점검한다.
- 다음 Phase 전에 분리 필요성을 검토한다.

파일 500줄 이상:
- 리뷰 부담이 커진 상태로 본다.
- 분리 계획을 세운다.

함수 50줄 이상:
- 여러 일을 하고 있는지 확인한다.
- 이름 붙일 수 있는 하위 함수가 있는지 검토한다.

동일한 로직 2회 이상 반복:
- 공통 함수 또는 service 분리 후보로 본다.
```

### 4-4. Phase Maintenance에서 허용되는 작업

```text
함수 이름 정리
작은 함수 추출
반복 코드 제거
route 파일 분리
template 일부 분리
테스트 이름 정리
문서 최신화
불필요한 주석 제거
```

### 4-5. 별도 승인 필요한 작업

```text
DB 스키마 변경
권한 정책 변경
보안 정책 완화
인증 흐름 변경
대규모 폴더 구조 변경
Repository 패턴 등 새 아키텍처 패턴 도입
기능 동작 변경이 섞인 리팩토링
```

## 5. AI와 Codex에 적용되는 원칙

AI와 Codex는 구조화를 제안할 수 있지만, 자동으로 대규모 구조화를 진행하지 않는다.

AI/Codex는 각 작업 전에 다음을 제시한다.

```text
이번 작업이 기능 추가인지 정리 작업인지 구분
구조화가 필요한 이유
변경 대상 파일
영향 범위
테스트 방법
사용자가 직접 확인해야 할 코드
```

Codex에게 금지되는 지시:

```text
프로젝트 구조 전체를 한 번에 고쳐라.
서비스/레포지토리/스키마 계층을 전부 만들어라.
알아서 좋은 구조로 바꿔라.
```

Codex에게 허용되는 지시:

```text
이 route 함수가 길어졌으니 함수 추출 후보를 제안해라.
이 파일에서 반복되는 검증 로직을 찾아라.
변경 전/후 최소 diff를 보여줘라.
테스트가 깨지지 않는 범위에서 작은 분리를 제안해라.
```

## 6. 수업 적용 방향

이 방식은 수업에서도 사용할 수 있다.

권장 수업 흐름:

```text
1. 한 파일로 작동하는 앱 만들기
2. 함수로 나누기
3. 파일로 나누기
4. 화면과 로직 분리하기
5. 데이터 저장 분리하기
6. 테스트 가능한 구조로 바꾸기
7. AI에게 리팩토링 제안받기
8. 사람이 제안을 검토하고 선택하기
```

학생들이 배워야 할 것은 다음이다.

```text
AI가 만든 결과를 무조건 수용하지 않는다.
왜 나누는지 설명할 수 있어야 한다.
구조는 처음부터 주어지는 것이 아니라 필요에 의해 생긴다.
테스트와 리뷰가 가능한 크기로 코드를 유지한다.
```

---

## 7. 최종 원칙

```text
빠르게 만드는 것보다 이해 가능한 속도로 만든다.
처음부터 완벽한 구조보다, 작동하는 코드에서 출발한다.
복잡해지는 순간을 놓치지 않고 최소 범위로 정리한다.
리팩토링은 미루지 않되 과하게 하지 않는다.
모든 구조 변경은 사용자가 이해하고 승인한 뒤 진행한다.
```
